# Phase 04-03 Plan - Asset Provenance And Protected Slot Actions

Status: planned

## Objective

Define and implement provenance-aware slot actions for core anime assets so operators can replace or remove media without breaking later automated refresh behavior.

## Requirements

- `ASST-01` Core asset slots expose provenance-aware state
- `ASST-02` Operator can replace a slot explicitly
- `ASST-03` Operator can remove a slot explicitly
- `ASST-05` Manual replacement is protected from later automated overwrite
- `OWNR-02` Source of the current slot value is understandable
- `RLY-02` Safe fallback and error feedback exist

## Implementation Scope

### Asset Semantics

- Covered persisted image slots in this phase are `cover`, `banner`, and `backgrounds`.
- Store or derive whether the current slot value is provider-backed or manually replaced.
- Make manual replacement explicit in UI messaging.
- Background/theme videos remain provider-only and are excluded from local persistence.
- Persisted anime assets must have runtime priority over provider fallbacks.

### Data Model

- Reuse `media_assets` as the file/blob descriptor source-of-truth.
- Add anime-owned references for single slots:
  - `cover`
  - `banner`
- Add an anime-owned ordered relation for plural `backgrounds`.
- Store ownership at the anime-slot layer, not only in UI state:
  - `manual`
  - `provider`
- Preserve enough slot metadata to support:
  - explicit remove
  - manual replace
  - provider refresh
  - runtime precedence

### Runtime Read Path

- Public anime runtime must read persisted anime assets before consulting Jellyfin fallbacks.
- Manual persisted assets win over provider assets.
- Provider fallback is only used when the persisted slot is absent.
- Background videos remain resolved from Jellyfin only and are not mixed into persisted image-slot selection.

### Slot Actions

- Add explicit replace/remove actions for covered slots.
- Ensure later metadata resync does not silently overwrite manually replaced assets.
- Return clear no-op or blocked-state feedback when a slot cannot be refreshed from provider data.
- Allow provider-owned banner/background slots to refresh on explicit resync.
- Allow new provider-owned backgrounds to be appended on resync without displacing manual backgrounds.

### Resync Rules

- `manual` slot:
  - never auto-overwrite
  - stays active until explicit remove/replace
- `provider` slot:
  - may refresh on explicit Jellyfin metadata/asset resync
- `empty` slot:
  - may be filled from Jellyfin on explicit resync
- Background append behavior:
  - new provider backgrounds may be added
  - existing manual backgrounds must retain order and identity
  - provider refresh must not reorder or delete manual backgrounds silently

### Suggested Execution Slice

1. Schema and repository support for anime banner/background persistence
2. Runtime precedence update for public anime assets
3. Admin edit-route slot actions and provenance rendering
4. Resync integration for provider-owned banner/background slots

### UX Constraints

- Keep asset controls inside the existing editor experience.
- Use concise German operator text.
- Avoid building a generic asset gallery.

## Non-Goals

- video/theme asset persistence or upload
- multi-slot bulk actions
- public asset provenance display
- anime logo persistence

## Verification

- tests cover replace/remove behavior and protection from later automated overwrite
- manual smoke confirms protected manual asset state survives metadata resync
- repository tests cover ownership-aware slot writes and ordered background persistence
- runtime tests confirm persisted anime assets beat Jellyfin fallback on public reads
- edit-route smoke confirms:
  - manual banner survives Jellyfin resync
  - manual background survives Jellyfin resync
  - provider-owned background list can grow with new Jellyfin entries
