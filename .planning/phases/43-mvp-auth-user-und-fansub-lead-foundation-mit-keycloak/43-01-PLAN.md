<objective>
Build the first execution slice of the Keycloak auth foundation: local Keycloak stack, token-validation seam, `app_users` foundation, backchannel-logout invalidation, and the minimal bootstrap path that makes later platform-admin and fansub-lead assignment possible.

Purpose: Before Team4s can delegate fansub leadership or secure admin routes meaningfully, the project needs a real external identity provider plus a clean internal app-user model. This plan establishes the infrastructure and backend auth baseline only; full browser-login adoption and broader admin runtime wiring belong to the next plan.
</objective>

    - `keycloak`

    Requirements for this slice:
    - Keycloak uses its own PostgreSQL database and persistent volume
    - local config is environment-driven
    - local development may use `start-dev`
    - the setup is documented so a developer can reach the Keycloak Admin Console locally
    - the realm/client baseline is automated as far as practical through `infra/keycloak/realm-team4s.json` and/or `scripts/keycloak/bootstrap-keycloak.sh`
    - automation is idempotent enough for repeated local setup runs
    - only global Keycloak roles are created: `platform_admin`, `content_admin`, `user`
    - no fansub-domain roles such as `fansub_lead`, `editor`, `designer`, or `translator` are created in Keycloak
    - Keycloak and Team4s are prepared for backchannel logout in local development, including the app-side endpoint/config seam needed to invalidate local auth state

    Keep this infrastructure slice self-contained and do not yet mix in full app permission logic.

    Build the new identity seam in the backend.

    This slice must introduce:
    - migration(s) for `app_users`
    - migration(s) for `app_user_global_roles`
    - backend token validation against Keycloak issuer/JWKS
    - a minimal normalized authenticated actor seam resolved from Keycloak claims
    - auto-create pending `app_user` on first valid login
    - backend handling for Keycloak backchannel logout so local auth/session state is invalidated when the upstream session ends
    - a minimal authenticated read seam, preferably `/api/me`, sufficient to prove claim resolution and local app-user lookup

    Be explicit in code and docs that `app_users` is the new authenticated app principal.
    Do not silently overload legacy `users` as if it were the same thing.
    Also document the SQL bootstrap path for the first Team4s `platform_admin`; this remains an app-database concern even if Keycloak can seed a dev user.
