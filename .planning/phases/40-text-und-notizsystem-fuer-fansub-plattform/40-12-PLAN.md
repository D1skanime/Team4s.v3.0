## Goal

Die zentrale Domain-Luecke schliessen, dass `release_version_notes` aktuell jede beliebige `member_id` und `role_id` akzeptiert, solange die IDs technisch existieren. Nach diesem Plan muss der Server vor jedem Insert/Update beweisen, dass die Member/Rollen-Kombination wirklich zur konkreten Release-Version gehoert.

## Context

- `AGENTS.md` - Release-Kontext darf nicht an die falsche Domain-Entity haengen
- `40-CONTEXT.md` - beteiligte Member/Rollen kommen aus `release_member_roles` oder dem Editor-Context
- `release_version_notes_repository.go` - aktueller Bulk-Upsert ohne serverseitigen Contributor-Guard
- `admin_content_release_version_notes.go` - aktuelles Handler-Mapping
- `ReleaseVersionNotesTab.tsx` - Frontend sendet Bulk-Array und geht von serverseitiger Wahrheit aus

## Tasks

### Task 1: Contributor-Guard im Repository einziehen

In `backend/internal/repository/release_version_notes_repository.go`:

- vor dem eigentlichen Bulk-Upsert die gueltigen `(member_id, role_id)`-Paare fuer `releaseVersionID` laden
- jede Eingabenote dagegen validieren
- bei Abweichung einen expliziten Domain-Fehler zurueckgeben
- die Validierung muss fuer Insert und Update gleichermassen gelten
- wenn moeglich in derselben Transaktion bleiben, damit Load + Write konsistent sind

Nicht tun:

- keinen stillen Filter, der ungueltige Eintraege einfach ignoriert
- keine implizite "Reparatur" des Requests

### Task 2: Handler-Fehlermapping und Request-Vertrag

In `backend/internal/handlers/admin_content_release_version_notes.go`:

- invaliden Contributor-Kontext als 400 oder 409 mit klarer deutscher Meldung zurueckgeben
- Konflikt auf UNIQUE weiter getrennt behandeln
- darauf achten, dass die API fuer das Frontend unterscheidbar macht:
  - Duplicate Note
  - ungueltige Member/Rollen-Zuordnung
  - Note nicht gefunden

### Task 3: Backend-Tests

Neue Tests in `admin_content_release_version_notes_test.go`:

- gueltiger Bulk-Save geht durch
- ungueltige `member_id`
- ungueltige `role_id`
- gueltige IDs, aber Paar gehoert nicht zu dieser Release-Version
- 409 fuer echte Duplicate-Situation bleibt erhalten

### Task 4: Frontend-Regression

Die bestehenden Frontend-Tests fuer `ReleaseVersionNotesTab` gezielt erweitern:

- UI zeigt die neue fachliche Fehlermeldung verstaendlich an
- bisherige Bulk-Save-Tests fuer legitime Rollen bleiben gruen
