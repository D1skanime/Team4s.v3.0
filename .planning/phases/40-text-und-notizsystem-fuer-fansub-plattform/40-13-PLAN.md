## Goal

Die restlichen Vertragsprobleme von Phase 40 aufräumen: `member_group_stories` soll kein irreführendes Edit-Verhalten mehr haben, und die Delete-/Update-Pfade für Fansub-Notes sollen stärker an den konkreten Routen-Kontext gebunden werden, damit Cross-Entity-Fehlgriffe erschwert werden.

## Context

- Review-Findings aus Phase 40:
  - `role_id` ist optional, aber per PATCH praktisch nicht sauber auf `NULL` rücksetzbar
  - `member_id` ist im UI editierbar, wird serverseitig im Update aber ignoriert
  - mehrere Delete-Pfade löschen bisher effektiv nur über nackte `noteId`
- `40-CONTEXT.md` - `member_group_stories.role_id nullable`
- `NotesTab.tsx` - aktueller Story-Editor
- Handler-Split über `admin_content_anime_project_notes.go`, `admin_content_fansub_group_notes.go` und `admin_content_member_stories.go`

## Tasks

### Task 1: Entscheidung für role_id-Edit-Vertrag sauber ziehen

Triff nach kurzer Bestandsprüfung eine explizite Entscheidung und setze sie konsequent um:

Option A:

- `role_id` darf im Update wirklich auf `NULL` gesetzt werden
- Repository/Handler müssen das technisch eindeutig unterstützen

Option B:

- `role_id` darf im bestehenden Datenmodell nur beim Anlegen optional sein
- dann muss die UI das im Editiermodus ehrlich abbilden und keine scheinbare "Rolle entfernen"-Funktion mehr zeigen

Wichtig: Die Entscheidung in `40-VALIDATION.md` und ggf. `DECISIONS.md` dokumentieren, falls sie später wieder diskutiert werden könnte.

### Task 2: member_id-Vertrag bereinigen

In `NotesTab.tsx` und dem Story-API-Fluss:

- kein `memberId` mehr im Update-Payload senden, wenn der Server das Feld nicht ändert
- falls `memberId` fachlich unveränderlich ist: UI im Edit-Modus als read-only oder gar nicht editierbar
- wenn fachlich doch editierbar gewünscht: dann nur mit expliziter serverseitiger Unterstützung und passenden Guards

Für diesen Gap-Closure-Plan ist die bevorzugte Richtung: irreführendes Edit-Feld entfernen oder read-only machen, nicht unnötig neue Semantik einführen.

### Task 3: ID-Scoped Delete-/Update-Pfade nachziehen

Die Fansub-Note-Repositories und Handler prüfen und dort enger scopen, wo heute nur über nackte `id` gearbeitet wird:

- `fansub_group_notes`
- `member_group_stories`
- `anime_fansub_project_notes`

Bevorzugt:

- Handler lesen den Routen-Kontext komplett aus
- Repository-Methoden löschen/aktualisieren über `id + parent-context`

### Task 4: Tests und Validierungsdoku

- `NotesTab.test.tsx` für den bereinigten Story-Vertrag anpassen/erweitern
- Backend-Tests für scoping-sichere Delete-/Update-Pfade ergänzen, mindestens dort, wo neues Verhalten eingeführt wird
- `40-VALIDATION.md` aktualisieren, damit die neue Vertragslage nicht von der alten Dokumentation überdeckt wird
