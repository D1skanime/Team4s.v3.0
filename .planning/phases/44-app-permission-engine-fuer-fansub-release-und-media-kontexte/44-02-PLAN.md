Purpose: The engine only becomes valuable once real fansub/release/media surfaces route through it and the frontend can ask the backend what actions are allowed in a concrete context.
</objective>

  <action>
    Swap the current duplicated `requireAdmin(...)` usage on the prioritized surfaces for permission-engine-backed checks.

    Priority route families:
    - fansub group read/edit
    - fansub members view/manage
    - release read/create/update
    - release version read/create/update
    - release version media read/upload/update/delete
    - release/member-role/translation/qc description-like writes on the existing notes/description handlers

    Each handler should:
    - pass the narrowest available resource identifier into the permission engine
    - let the engine resolve the rest of the context
    - never trust frontend-supplied ownership or group context over DB truth
    - return 403 when the actor is authenticated but not allowed
    - reuse existing handler/repository seams where they already model the resource correctly instead of introducing handler-local permission joins
