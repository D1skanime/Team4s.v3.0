# Phase 20.1: DB Schema v2 Physical Cutover - Context

## Trigger

The current repo has a canonical target schema in `docs/architecture/db-schema-v2.md`, but feature work has still been able to drift into partial/legacy persistence. Phase 18/19 proved this with episode import: the UI became usable, yet imported media still ended up in the old `episode_versions` model.

The user wants the architecture schema built as real database tables first, then feature work can use that foundation instead of inventing more partial structures.

## Scope

Phase 20.1 is a database foundation phase.

It must:

- Treat `docs/architecture/db-schema-v2.md` as the authoritative target.
- Build or reconcile every currently documented target table, lookup table, join table, foreign key, unique constraint, and index.
- Include the newly documented filler metadata model:
  - `Episode.filler_type_id`
  - `Episode.filler_source`
  - `Episode.filler_note`
  - `EpisodeFillerType`
- Include normalized file-to-episode coverage:
  - `ReleaseVariantEpisode`
- Remove the legacy episode-version table family after local test data is reset:
  - `episode_version_episodes`
  - `episode_version_images`
  - `episode_versions`
- Update schema-facing code/tests enough that the app no longer depends on those dropped tables.

## Explicit Deletion Decision

The project currently only has test anime data in the old episode import tables. The user explicitly approved deleting old episode tables instead of preserving them.

Important boundary:

- Delete legacy release-bearing episode tables: `episode_versions`, `episode_version_images`, `episode_version_episodes`.
- Do not delete target identity tables such as `episodes`, `episode_titles`, `episode_types`, or `languages`.
- Do not delete anime/create asset data outside the controlled local reset unless the execution plan explicitly scopes it to local dev verification.

## Why This Comes Before Phase 20

Phase 20 was planned to make episode import write the release graph. That is still correct, but it should run after Phase 20.1.

The new order is:

1. Phase 20.1: physicalize DB Schema v2 and remove old episode-version persistence.
2. Phase 20: implement/import on top of the new schema.

This prevents another cycle where a frontend/backend feature quietly reintroduces legacy slot-specific or flat episode-version behavior.

## Existing Legacy Tables To Remove

Known legacy table family from migrations:

- `0012_episode_versions.up.sql` creates `episode_versions`.
- `0018_episode_version_images.up.sql` creates `episode_version_images`.
- `0043_add_episode_version_episodes.up.sql` creates `episode_version_episodes`.

Phase 20.1 must also search the codebase for references to these names and either migrate them to release-native tables or mark them as deleted compatibility only where unavoidable.

## Target Verification

Phase 20.1 is complete when:

- A clean local DB migrates from zero to the full target schema.
- Schema audit confirms every documented table/column/constraint/index exists.
- Legacy episode-version tables are absent.
- Backend/frontend tests no longer require the dropped tables.
- Docker can rebuild against the schema.
