## Trust Boundaries

| Grenze | Beschreibung |
|--------|-------------|
| Frontend FormData → Backend Handler | visibility_code/review_status_code kommen als nicht-vertrauenswürdige Strings |
| Backend Handler → Repository | Codes werden per DB-Lookup aufgelöst — kein ID-Trust vom Frontend |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-79-02-01 | Tampering | visibility_code/review_status_code in FormData | mitigate | Whitelist-Validierung im Handler vor Weiterleitung (valide Codes aus Migration 0037/0097); ungültige Codes → badRequest 400 |
| T-79-02-02 | Broken Access Control (Lock I) | Upload-Handler schreibt unter Owner-Kontext | mitigate | Existierende CanForFansubGroup / CanForReleaseVersion Capability-Gates bleiben unverändert; Phase 79 fügt nur Status-Felder hinzu, verändert keine Capability-Prüfung |
| T-79-02-03 | Elevation of Privilege | Sofort-Freigabe-Pfad (D-09) per manipuliertem reviewStatusCode='approved' | mitigate | Branding-Handler (fansub_media_upload.go, theme_assets) überschreiben mit hartem Default 'public'/'approved'; ein bösartiger approved-Wert ändert für Branding-Surfaces nichts (Default gleich). Für Prozessmedien-Handler (RVM): wenn Frontend 'approved' schickt, nimmt Backend es an — serverseitige Leader-Capability-Prüfung bleibt Guard; explizite Freigabe via Review-Workflow (D-04) |
| T-79-02-04 | Tampering | Sub-SELECT mit nicht-existentem Code | mitigate | LIMIT 1 gibt 0 rows zurück → NULL-ID → INSERT schlägt fehl wenn visibility_id/review_status_id NOT NULL-Constraint besteht. Spalten sind nullable (Migration 0097) — als weitere Härtung: Whitelist im Handler vor DB-Aufruf (T-79-02-01 abgedeckt) |
| T-79-02-SC | Tampering | npm/pip/cargo installs | accept | Keine neuen Pakete — nur bestehender Go/TS-Stack |
