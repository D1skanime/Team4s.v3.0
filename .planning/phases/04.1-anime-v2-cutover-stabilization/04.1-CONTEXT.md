# Phase 04.1: Anime v2 Cutover Stabilization - Context

**Gathered:** 2026-03-31
**Status:** Ready for planning
**Source:** Runtime handoff + current local implementation state

<domain>
## Phase Boundary

This inserted phase stabilizes the active anime v2 runtime slice before broader provenance work continues.

The phase is specifically about the currently live create/edit/read/admin behavior on `team4s_v2`, not about designing the full provenance UI or the later relation workflow. The immediate need is to stop the admin edit surface from drifting between legacy flat-column assumptions and the new normalized v2 schema.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- Treat the current anime v2 cutover as a dedicated stabilization phase instead of silently folding it into broad Phase 04 provenance work.
- Preserve the shared admin anime editor surface from Phase 1 instead of introducing a second temporary edit workflow.
- Keep the backend runtime on `team4s_v2`; do not revert the local stack back to the legacy anime schema.
- Cover upload in local development may still write to `frontend/public/covers`, but the active anime record must also persist the chosen cover through the v2 data model so later reads show the same cover again.
- Manual create and edit must keep Jellyfin linkage intact when `source` is present, even if the operator replaces the cover or edits metadata.
- Error responses for admin anime save/load failures must include operator-usable context so a generic `interner serverfehler` is no longer the only visible signal.
- Favor v2-native repository reads and writes over expanding legacy compatibility columns further.

### The Agent's Discretion
- Whether the remaining update migration is delivered as one plan or split into multiple small plans.
- Whether validation should lean more on schema-aware shims or direct v2-only write paths, as long as the result is stable against the current runtime schema.
- Whether lightweight docs/handoff updates belong in the same execution wave or in a small trailing plan.

</decisions>

<specifics>
## Specific Ideas

- The plan should explicitly account for the fact that some local databases may temporarily have `slug` but not all runtime columns such as `source`, `status`, or `max_episodes`.
- Cover persistence needs end-to-end verification: upload file exists, anime record returns that cover on next read, and the edit page shows the same value after refresh.
- Source persistence needs end-to-end verification: create/edit a Jellyfin-linked anime, then confirm preview/resync/media lookup still see the linkage.
- Error-output work should be treated as part of stabilization, not as optional polish.

</specifics>
