| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none - existing manual CSS-module admin system |
| Icon library | lucide-react |
| Font | existing app sans-serif stack |

**Pre-populated from upstream**
- CONTEXT.md: shared draft create surface remains the base, anime-name is the anchor, Jellyfin stays preview-only until final save, compact-first source selection, strong card-based candidate review, advisory type suggestion with visible reasoning.
- RESEARCH.md: Next.js + React + CSS modules remain standard; reuse the shared `/admin/anime/create` shell and current save-bar contract.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge spacing, chip internals |

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Rules:
- Use only weights `400` and `600`.
- Metadata lines such as Jellyfin ID and full path use 14px body sizing, not 12px microcopy.
- Truncate only in the compact picker. In candidate review cards and draft context, full path must wrap and remain readable.

Accent reserved for: `Jellyfin Sync` primary action when enabled, selected candidate state, active step/current review state, "prefilled from Jellyfin" badges, retry actions, and high-confidence type suggestion chips. Never use accent for all inputs, all links, or passive metadata.

## Phase Surface Contract

### Entry and Anchor
- The existing shared create surface remains the base screen. Do not introduce a separate Jellyfin wizard route.
- The anime-name field is the first visual anchor in the form.
- `Jellyfin Sync` and `AniSearch Sync` sit immediately adjacent to the anime-name input in the same horizontal cluster on desktop and stack directly beneath it on mobile.
- Both buttons stay disabled until the title contains meaningful input. Minimum contract: trimmed value length >= 2 and not only punctuation.
- Disabled buttons must still show helper text explaining why they are disabled.

### Compact-First Source Selection
- The first Jellyfin interaction is compact: dropdown, command-style result popover, or compact chooser attached to the title area.
- The compact chooser must privilege folder/order-name matching and show stronger matches first.
- Compact results may show only title, year, and one-line path preview, but selection from this list must open the full candidate review state before draft hydration.
- Do not jump straight from the first compact list into saveable draft state without an explicit review step.

### Candidate Review Cards
- Candidate review is the visual centerpiece of this phase and must feel stronger than the compact picker.
- Present candidates as vertically stacked cards or a 2-column card grid on wide desktop. Never use a plain table.
- Each card must show:
  - Title
  - Jellyfin ID
  - Full path
  - Library or parent context
  - Type hint
  - Poster preview
  - Banner preview
  - Logo preview
  - Background preview
- Image previews should use fixed-ratio tiles so cards stay stable while loading.
- Full path and parent/library context must remain visible without hover.
- Exact or stronger matches receive a visible confidence badge, but the operator still chooses manually.
- Candidate cards must not contain asset deselection controls.

### Selection and Draft Handoff
- After the operator selects a candidate, open the same shared draft immediately in-place or via the same create route, already prefilled.
- The save bar remains the only persistence control. No card-level save, import, or hidden persistence is allowed.
- Prefilled fields and asset slots should show subtle source badges indicating they came from Jellyfin preview data.
- Asset deselection happens only inside the draft after handoff.

### Type Suggestion Contract
- Jellyfin-derived type is advisory only.
- Show the suggested type next to the editable field, not instead of it.
- Visible reasoning is required and must reference path/name evidence in plain language.
- Confidence tone may be shown as `hoch`, `mittel`, or `niedrig`, but low confidence must never block editing.

## Interaction and State Rules

- Loading state for `Jellyfin Sync`: keep button width stable, replace label with spinner + progress copy.
- Candidate card selection: selected card gets accent border, elevated surface, and explicit `Ausgewaehlt` badge.
- Hover state is secondary only. All critical states must be visible without hover.
- Error feedback appears inline near the affected stage and also in the shared page feedback area when the whole fetch fails.
- Empty search results must offer a clear retry path and suggest checking folder/order naming.
- The operator must always have a visible path back to manual editing without losing control of the unsaved draft.
- Motion is restrained: 150-200ms opacity/transform transitions only for popover opening, card selection, and draft hydration confirmation.
- Respect reduced motion; disable non-essential transitions.

---

## Layout Contract

- Desktop >= 1200px:
  - Title field and source actions form one compact top row.
  - Candidate review uses strong card layout with media previews visible above the fold.
  - Draft handoff keeps the existing editor shell and sticky save bar.
- Tablet 768-1199px:
  - Title row may wrap to two lines.
  - Candidate cards collapse to one column if previews become cramped.
- Mobile < 768px:
  - Title input first, then full-width `Jellyfin Sync`, then full-width `AniSearch Sync`.
  - Candidate cards remain single-column.
  - Preview media stack vertically in this order: poster, banner, logo, background.

---

## Accessibility and Operator-Safety

- Minimum touch target: 44px.
- Focus order must follow visual order: title -> Jellyfin Sync -> AniSearch Sync -> compact results -> candidate cards -> draft fields -> save bar.
- Icon-only controls are not allowed in the candidate review surface.
- All previews need meaningful alt text, including asset type and candidate title.
- Use color plus text for status; confidence and warning states cannot rely on color alone.
- Operator-safe tone means no ambiguous actions like `Importieren` before the preview step is complete.
