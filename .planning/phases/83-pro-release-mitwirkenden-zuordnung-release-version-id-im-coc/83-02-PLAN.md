Neue Methode `ListActorContributionRolesForVersion` an `AuthzRepository` anhängen. Die Methode folgt dem Muster von `ListActorGroupRoles` (gleiche Error-Handling-Struktur, gleicher Rows-Loop).

Implementierung in zwei Schritten:

Schritt 1 — Versions-spezifische Query (release_version_id = $versionID):
```sql
SELECT DISTINCT acr.role_code
FROM anime_contributions ac
JOIN anime_contribution_roles acr ON acr.anime_contribution_id = ac.id
JOIN members m ON m.id = ac.member_id
JOIN fansub_group_members fgm ON fgm.member_id = m.id
  AND fgm.fansub_group_id = ac.fansub_group_id
  AND fgm.app_user_id = $appUserID
WHERE ac.release_version_id = $versionID
  AND fgm.status = 'active'
```

Schritt 2 — Nur ausführen wenn Schritt 1 keine Ergebnisse liefert (Fallback anime-weit):
```sql
SELECT DISTINCT acr.role_code
FROM anime_contributions ac
JOIN anime_contribution_roles acr ON acr.anime_contribution_id = ac.id
JOIN members m ON m.id = ac.member_id
JOIN fansub_group_members fgm ON fgm.member_id = m.id
  AND fgm.fansub_group_id = ac.fansub_group_id
  AND fgm.app_user_id = $appUserID
JOIN release_versions rv ON rv.id = $versionID
JOIN fansub_releases fr ON fr.id = rv.release_id
WHERE ac.release_version_id IS NULL
  AND ac.anime_id = fr.anime_id
  AND ac.fansub_group_id IN (
      SELECT fansub_group_id FROM release_version_groups WHERE release_version_id = $versionID
  )
  AND fgm.status = 'active'
```

Defensiv-Check am Anfang: `appUserID <= 0 || releaseVersionID <= 0` → return nil, nil.

Error-Format: `fmt.Errorf("list actor contribution roles version=%d user=%d step=%d: %w", releaseVersionID, appUserID, step, err)`.

WICHTIG: `ListActorGroupRoles` wird NICHT entfernt — weiterhin für Leader-Check in permissions.go benötigt.

Der `fansub_group_id`-Scope im Join auf `fansub_group_members` begrenzt Zugriff auf korrekte Gruppe (Security: verhindert Cross-Gruppen-Leckage, T-83-01).

**Änderung 2: CanForReleaseVersion umbauen**
Die aktuelle Implementierung (Delegation an `canForContext`) ersetzen durch eigenständige Methode:

Ablauf (strikt in dieser Reihenfolge — Pitfall 1 aus RESEARCH.md):
1. Basis-Checks: `s == nil || s.resolver == nil` → deny(ReasonUnauthorized); `actor.AppUserID <= 0` → deny; `actor.Status == "disabled"` → deny(ReasonDisabledUser); `actor.IsPlatformAdmin` → allow(ReasonPlatformAdmin)
2. `ResolveReleaseVersion(ctx, releaseVersionID)` → resourceCtx; nil-Prüfung → deny(ReasonResourceNotFound)
3. **Leader-Check ZUERST (D-05)**: Für jede `fansubGroupID` in `resourceCtx.FansubGroupIDs`: `ListActorGroupRoles(ctx, actor.AppUserID, fansubGroupID)` → hat role `fansub_lead` oder `project_lead` UND `roleMatrix[role]` enthält action? → allow immediately
4. **Contribution-Check (D-01..D-04)**: `ListActorContributionRolesForVersion(ctx, actor.AppUserID, releaseVersionID)` → roleCodes; falls ein roleCode in `roleMatrix[code]` die action enthält → allow; falls roleCodes nicht leer aber keine action erlaubt → deny(ReasonInsufficientRole); falls roleCodes leer → deny(ReasonNoMembership)

`roleAllows`-Hilfsfunktion (bereits vorhanden) weiterverwenden. `denied`-Hilfsfunktion weiterverwenden. `canForContext` bleibt für andere Methoden erhalten — NICHT entfernen.

KRITISCH: Leader-Bypass (Schritt 3) MUSS vor Contribution-Check (Schritt 4) liegen. Ein `fansub_lead` hat oft selbst keine Contribution — würde sonst fälschlicherweise abgelehnt.
