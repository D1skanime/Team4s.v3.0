<task type="auto">
  <name>Task 2: Implement invitation create, list, cancel, and accept endpoints</name>
  <read_first>
    - backend/internal/handlers
    - backend/internal/repository
    - Phase-45 member-management runtime seam
  </read_first>
  <action>
    Implement the backend endpoints for:
    - creating an invitation for a fansub group
    - listing open or relevant invitations for a fansub group
    - cancelling an invitation
    - accepting an invitation as the current authenticated app user

    Requirements:
    - permission-engine enforcement for create/view/cancel
    - authenticated accept flow
    - no plaintext token persistence
    - expiry and terminal status enforcement
    - role validation through the central role list
    - acceptance routed through the canonical Phase-45 membership path
    - create rejects unknown role codes
    - create requires at least one role code
    - create rejects duplicate pending invitation for the same group + email
    - create rejects already-active group members
    - list responses never return token hashes or raw tokens
    - cancel only works for pending invitations belonging to the addressed group
    - accept hashes the provided token and matches it against `token_hash`
    - accept validates invitation status, expiry, email match, and membership conflict rules

    Expected endpoint shapes:
    - `POST /api/admin/fansub-groups/:id/invitations`
    - `GET /api/admin/fansub-groups/:id/invitations`
    - `POST /api/admin/fansub-groups/:id/invitations/:invitationId/cancel`
    - `POST /api/invitations/accept`

    Expected create response shape:
    - `id`
    - `email`
    - `status`
    - `expiresAt`
    - `inviteLink` or equivalent one-time token carrier

    Expected accept response shape:
    - `accepted`
    - `fansubGroupId`
  </action>
  <verify>
    <automated>cd C:/Users/admin/Documents/Team4s/backend && go test ./internal/handlers/... -run "Test.*Invitation|Test.*Accept"</automated>
  </verify>
  <done>
    - The invitation lifecycle is complete and backend-enforced.
    - Acceptance writes to the canonical membership model rather than any legacy seam.
  </done>
</task>
