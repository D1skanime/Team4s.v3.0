<discovery>

## Ist-Zustand

- Route: the runtime UI currently lives at `frontend/src/app/admin/profile/page.tsx` as `AdminProfilePage`. There is no `frontend/src/app/me/profile/page.tsx`.
- Existing non-admin route pattern: `frontend/src/app/manage/groups/page.tsx` already re-exports `../../admin/my-groups/page`, so transition routes are acceptable if ownership is explicit.
- API endpoints: backend routes exist for `GET /api/v1/me/profile`, `PUT /api/v1/me/profile`, and `POST /api/v1/me/profile/avatar`; frontend helpers are `getOwnProfile`, `updateOwnProfile`, and `uploadOwnProfileAvatar`.
- Data sources: `MemberProfileRepository` reads `app_users`, `app_user_global_roles`, `members`, `group_members`, `fansub_group_members`, `fansub_group_member_roles`, `release_member_roles`, `release_versions`, `release_version_groups`, `fansub_groups`, `contributor_roles`, `media_assets`, and `media_files`.
- Current UI issues: technical account fields are prominent, labels still show raw codes in places, active dates are free numeric inputs, visibility is a select, memberships/credits are debug-like lists, and the page imports admin styles directly.
- Current GDS pattern: `frontend/src/app/admin/my-groups/page.tsx` already uses `Button`, `Card`, `Badge`, `EmptyState`, `ErrorState`, `LoadingState`, `PageHeader`, `SectionHeader`, and `Toolbar`; `/admin/profile` should move toward that global component language.

## Zielarchitektur

- Target route: `/me/profile`.
- Transition route: `/admin/profile` redirects to `/me/profile` or internally re-exports the same `MyProfilePage` without keeping a separate implementation.
- Component direction: `MyProfilePage`, `MemberProfileHero`, `ProfileBasicsForm`, `MembershipList`, `MembershipCard`, `ContributionSummaryCard`, `AccountSecurityCard`, `RoleChip`, `StatusBadge`.
- Data flow: 53A consumes the existing `GET /api/v1/me/profile` aggregate and existing mutation helpers. It must not invent fields when the DTO lacks them.
- Layout: profile hero first, then main column for basis/story/memberships/contributions and side column for avatar/visibility/account. In 53A the avatar/visibility cards may be visual organization only; deeper behavior lands in 53B.

## Datenquellen-Mapping

- Profil-Hero: `display_name`, `fansub_name`, `bio`, `avatar`, `active_from_year`, `is_currently_active`, `memberships`, `account_global_roles`; missing grouped role labels and display-ready group/logo data.
- Basisdaten: `display_name`, `fansub_name`, `bio`, `active_from_year`, `active_until_year`, `is_currently_active`; in 53A use constrained year/checkbox/select controls only where current contract supports them.
- Avatar: `avatar.public_url` via `members.avatar_media_id -> media_assets/media_files`; 53A only displays current avatar and the existing upload entry point if already present.
- Sichtbarkeit: `profile_visibility` supports `members_only | public`; 53A must not show a working third option unless the contract changes.
- Account & Sicherheit: `email`, `account_display_name`, `account_status`, `account_global_roles`, `keycloak_account_url`, `capabilities.can_open_keycloak_account`; all read-only.
- Mitgliedschaften: `memberships` from `group_members`, `fansub_group_members`, `fansub_group_member_roles`, `fansub_groups`; group logo is not currently in the profile DTO.
- Meine Beiträge: `historical_credits` is aggregated by group/role/release count; detailed Anime/Episode/Release-Version records are not yet available.
- Rich-Text-Bio: current profile page converts editor state to plain text; 53A may preserve existing behavior but must not claim rich-text persistence.

## Contract-/Backend-Lücken

- `shared/contracts/openapi.yaml` does not document the existing `/api/v1/me/profile` runtime API.
- `ProfileVisibility` only knows `public | members_only`.
- Activity period is year-only, not month/year.
- Profile rich text is plain-text converted.
- Avatar has no crop contract or variants.
- Group logo is missing in `MemberProfileMembership`.
- Contributions are aggregate-only, not paginated detail rows.
