<objective>
Phase 53A turns the current admin-looking profile screen into the first usable `/me/profile` Member Identity Hub foundation: correct route ownership, real data sources, role-neutral component naming, GDS-based layout, and honest membership/contribution display without changing deeper avatar, rich-text, visibility, or month-year contracts yet.
</objective>

<decision_coverage>

- D-01: `/me/profile` is the target route for all signed-in users.
- D-02: `/admin/profile` remains only a redirect or internal re-export, not a separate admin profile world.
- Execution decision for D-02: use internal re-export/transition wrapper by default; redirect requires a documented blocker and updated tests.
- D-03: Components and visible naming must be role-neutral.
- D-04: The UI must not feel like an admin-only backend form for normal members.
- D-05: Keycloak owns login, e-mail, password, MFA, technical account security, and technical identity.
- D-06: Team4s owns fansub profile data, avatar, bio/story, activity period, visibility, groups, roles, and contributions.
- D-07: Frontend code must not read tokens or call Keycloak helpers directly.
- D-08: Read-only account data must stay out of public-profile-ready components.
- D-09: OpenAPI documentation for own-profile read/update/avatar is mandatory.
- D-10: Runtime DTOs, frontend DTOs, API helpers, and OpenAPI move together.
- D-11: UI code must not infer undocumented response fields, status paths, or fallbacks.
- D-12: Missing or unclear visibility is not public.
- D-18: Role kinds remain separate.
- D-19: Technical role/status codes get German labels before user-facing display.
- D-20: Historical credits never grant permissions.
- D-21: Contributions stay honest aggregate/summary data unless a paginated contract is added.
- D-30: The screenshot is design direction, not pixel-perfect spec.
- D-31: The screenshot's information architecture guides the profile hub layout.
- D-32: No profile-only hardcoded shell/sidebar.
- D-33: Existing GDS components are used before page-local controls.
- D-34: German UI text uses correct umlauts.
- D-35: Save is active only for dirty Team4s profile fields.
- D-36: Keycloak-return refresh must not overwrite dirty Team4s profile fields.
- D-38: Aggregate profile reads must not pretend to have section-level partial failure boundaries.

</decision_coverage>

## Datenquellen-Mapping

- Profil-Hero: `display_name`, `fansub_name`, `bio`, `avatar`, `active_from_year`, `is_currently_active`, `memberships`, `account_global_roles`; missing grouped role labels and display-ready group/logo data.
- Basisdaten: `display_name`, `fansub_name`, `bio`, `active_from_year`, `active_until_year`, `is_currently_active`; in 53A use constrained year/checkbox/select controls only where current contract supports them.
- Avatar: `avatar.public_url` via `members.avatar_media_id -> media_assets/media_files`; 53A only displays current avatar and the existing upload entry point if already present.
- Sichtbarkeit: `profile_visibility` supports `members_only | public`; 53A must not show a working third option unless the contract changes.
- Account & Sicherheit: `email`, `account_display_name`, `account_status`, `account_global_roles`, `keycloak_account_url`, `capabilities.can_open_keycloak_account`; all read-only.
- Mitgliedschaften: `memberships` from `group_members`, `fansub_group_members`, `fansub_group_member_roles`, `fansub_groups`; group logo is not currently in the profile DTO.
- Meine Beiträge: `historical_credits` is aggregated by group/role/release count; detailed Anime/Episode/Release-Version records are not yet available.
- Rich-Text-Bio: current profile page converts editor state to plain text; 53A may preserve existing behavior but must not claim rich-text persistence.

## Contract-/Backend-Lücken

- `shared/contracts/openapi.yaml` does not document the existing `/api/v1/me/profile` runtime API; Phase 53 must add GET/PUT profile and POST avatar documentation.
- `ProfileVisibility` only knows `public | members_only`.
- Activity period is year-only, not month/year.
- Profile rich text is plain-text converted.
- Avatar has no crop contract or variants.
- Group logo is missing in `MemberProfileMembership`.
- Contributions are aggregate-only, not paginated detail rows.

## Risiken / offene Fragen

- Sidebar/App-Shell reuse must be verified before adding route UI. If no global shell exists for `/me/*`, extend the existing shell seam or keep the current app layout; do not hardcode a profile-only sidebar.
- Membership and contribution sections must respect `can_view_memberships` and `can_view_historical_credits`; if capabilities are false or missing, render a conservative scoped state.
- Group logos and contribution detail rows are tempting because the screenshot shows richer cards, but they are not available in the current profile DTO. 53A may show fallback icons/summary only unless the contract is expanded.
- `/admin/profile` should stay test-friendly by re-exporting or wrapping `/me/profile`; a redirect is only acceptable if a framework blocker is documented and tests are updated accordingly.
- The current page already contains Keycloak-return dirty-state behavior from Phase 52; recomposition must preserve it.
- `bio` and `member_story` are separate profile concepts. 53A keeps `bio` as the short card/hero description and keeps `member_story` visible/editable in a separate `ProfileStoryCard` using the current plain-text persistence path.
- CSS ownership moves with the route: `/me/profile` owns its profile layout CSS under `/me/profile`. It must not import `admin/profile/page.module.css`; `/admin/profile` should only re-export or wrap.
