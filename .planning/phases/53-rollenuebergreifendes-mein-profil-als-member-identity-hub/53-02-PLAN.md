<discovery>

## 53B Ausgangslage

- Avatar: existing profile upload accepts one file via `POST /api/v1/me/profile/avatar`; backend validates image MIME through shared image rules but currently has no profile-specific 5 MB limit, crop payload, remove endpoint, or generated variants.
- Crop analog: `frontend/src/components/admin/MediaUpload.tsx` already contains project-owned crop UI, keyboard support, focus handling, and crop math helpers for group logo media. Prefer extracting/reusing focused primitives instead of adding `react-easy-crop` unless extraction is clearly worse.
- Rich text: `RichTextEditor` exists, but the profile page currently converts editor JSON to plain text before saving `member_story`. Backend `TipTapService` can validate JSON, render sanitized HTML, and extract text, but the profile API is not wired to it.
- Visibility: profile visibility has only `members_only` and `public`. The requested `Für Gruppen sichtbar` value needs a contract/DB decision before it is enabled.
- Activity period: profile fields are `active_from_year` and `active_until_year`. Month/year must not be derived from memberships or credits.
- Contributions: current `historical_credits` are aggregate-only. A detailed, paginated contributions endpoint does not exist.
- Membership card richness: current `MemberProfileMembership` lacks group logo data, so richer cards require a DTO/contract extension or fallback icons.
- OpenAPI: the existing own-profile endpoints are not documented in `shared/contracts/openapi.yaml`; 53A should add the baseline, and 53B must expand it for any avatar/rich-text/visibility/month-year/detail changes.

## 53B Target Contract Direction

- Avatar crop can be client-cropped to a safe raster file before upload, or backend can accept explicit crop metadata and generate variants. The plan must pick one contract and document why.
- Avatar validation must explicitly allow JPG, PNG, WEBP; reject SVG; enforce the profile-specific size limit; and keep server validation authoritative.
- Rich text should store TipTap JSON plus sanitized HTML/text only after validating allowed nodes/marks/links. If migration is required, add a new migration rather than editing historical migrations.
- Visibility should either stay limited to current values with clear UI copy, or add a third durable value such as `groups_visible` with defaults and public-query behavior. Do not show a fake working option.
- Month/year should be represented by explicit contract fields or a documented date encoding, not by guessing from group membership dates.
- Contributions should remain aggregate-only unless 53B explicitly adds a paginated detail endpoint and contract. Do not fake Anime/Episode/Release-Version rows from summary counts.

## Risiken / offene Fragen

- A three-option visibility UI (`Nur intern`, `Für Gruppen sichtbar`, `Öffentlich sichtbar`) matches the screenshot direction but cannot be fully interactive until the backend and public-query semantics support the third value.
- Month/year activity may need a migration. If no migration is added, the UI must stay year-only and clearly constrained.
- Rich text may require new durable fields for TipTap JSON and sanitized HTML/text. If those fields are not added, the editor must not pretend rich formatting persists.
- Avatar variants should reuse `media_files.variant`; if derivative generation is too large for this phase, document the follow-up instead of inventing a parallel avatar table.
- Partial failures depend on API shape. If `/me/profile` remains one aggregate response, section-level partial failures may only apply to separate avatar/upload/save actions, not initial membership/contribution reads.

## Security / Privacy Invariants

- Account data never goes into public-ready components.
- Public-profile-ready components receive only public-safe fields.
- Rich text never renders unsanitized HTML and never allows scripts, iframes, inline event handlers, or unsafe links.
- Historical credits remain read-only display data and do not grant capabilities.
- Frontend UI may use backend capability booleans, but backend remains authoritative for mutations.
