# Phase 52 Research: Profile Account Return Refresh Flow

## Current Code Findings

- `frontend/src/app/admin/profile/page.tsx` already renders the Keycloak account link with `target="_blank"` and `rel="noreferrer"` when `profile.capabilities.can_open_keycloak_account` and `profile.keycloak_account_url` are present.
- The same page loads the profile only inside the initial `useEffect()` that depends on `hasAccessToken` and `isClientInitialized`.
- There is no `window.focus`, `visibilitychange`, or account-handoff state in the profile page.
- `frontend/src/lib/api.ts` already exposes `refreshActiveAuthSession()`. In Keycloak mode it refreshes the runtime session, calls `/api/v1/me` with the fresh access token, and persists the resolved auth session.
- Backend `/api/v1/me` resolution goes through `AppAuthRepository.EnsureAppUserForIdentity()`, which updates `app_users.email`, `display_name`, `preferred_username`, `given_name`, and `family_name` from the current Keycloak identity.
- `MemberProfileRepository.GetOwnProfile()` reads `email` and `account_display_name` from `app_users`, so a post-refresh `getOwnProfile()` can display updated Keycloak account data without a new backend endpoint.
- Existing profile tests mock `getOwnProfile`, `updateOwnProfile`, and `uploadOwnProfileAvatar` but do not mock or assert `refreshActiveAuthSession`.

## Reuse Strategy

- Reuse `refreshActiveAuthSession()` instead of adding profile-local auth refresh logic.
- Reuse `getOwnProfile()` for account cards and profile payload.
- Reuse the existing success/error panel pattern on `page.tsx`; no new toast system is required.
- Keep the Keycloak link generated by the backend profile capability/URL response.
- Keep profile page local styles unless an existing global UI component is already used in this page.

## Implementation Notes For Executors

- Extract a local `loadProfile({ syncForm })` helper or equivalent inside `AdminProfilePage` so initial load can sync the form while return refresh can preserve dirty form state.
- Track dirty form state with explicit setters or a helper wrapper around `setForm`.
- Track whether Keycloak account management was opened from this page. Only then should return focus trigger account refresh.
- Use a small account snapshot function comparing `account_display_name`, `email`, `account_status`, and joined `account_global_roles`.
- Guard concurrent return refresh with `isRefreshingAccount` or a ref to avoid duplicate calls from focus and visibility events.
- Tests can simulate `window.dispatchEvent(new Event('focus'))` or `document.dispatchEvent(new Event('visibilitychange'))` after clicking the account link.
